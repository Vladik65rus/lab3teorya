## Разработка грамматики
### Терминалы (V_T):
`fun, return, Int, (, ), {, }, ;, :, ,, +, -, *, /, id, num`

### Полное определение грамматики в нотации БНФ:

### Грамматика в компактной форме (используется в реализации):

Где:
- `<S> `          - начальный символ (функция)
- `<P> `          - список параметров
- `<T>`          - тип данных
- `<E>`           - арифметическое выражение
- ` <O> `          - операнд
- `<Знак_сложения>` - + | -
- `<Знак_умножения>` - * | /

## Классификация грамматики по Хомскому

Данная грамматика относится к **контекстно-свободной грамматике (КС-грамматике, тип 2)** по классификации Хомского.

### Обоснование:

1. Все правила имеют вид `A → α`, где `A` — нетерминал, `α` — строка из терминалов и нетерминалов
2. Левая часть каждого правила состоит ровно из одного нетерминала
3. Правила не зависят от контекста (окружения)
4. Грамматика описывает вложенные структуры (например, выражения со скобками)

### Примеры правил, доказывающих КС-грамматику:

| Правило | Обоснование |
|---------|-------------|
| ` <S> `   → fun id ( <P> ) : <T> { return <E> } ;
| ` <P>  `  → <P_хвост> | ε
| ` <P_хвост>` → id : <T> | id : <T> , <P_хвост>
| ` <T>`    → Int
| ` <E> `   → <O> | <E> <Знак_сложения> <O>
| ` <O> `   → id | num | ( <E> ) | <O> <Знак_умножения> <O>
|  `<Знак_сложения>` → + | -
|  `<Знак_умножения>` → * | /

Грамматика **не является регулярной (тип 3)** из-за наличия рекурсивных правил и вложенных конструкций. В регулярных грамматиках все правила должны иметь вид `A → aB` или `A → a`, что не выполняется для правил вида `E → ( E )` и `E → E + E`.
### Грамматика в компактной форме (EBNF)
`S ` → fun id ( P ) : T { return E } ;
`P`  → id : T { , id : T }
`T ` → Int
`E ` → O { (+|-) O }
`O ` → id | num | ( E ) | O { (*|/) O }
## Классификация грамматики по Хомскому
### Обоснование:
|Критерий |	Выполнение | Пояснение |
|---------|------------|----------|
|Левая часть состоит из одного нетерминала|	✓	| Все правила имеют вид A → α |
|Отсутствие контекстной зависимости|	✓	|Правила не зависят от окружения|
|Наличие рекурсивных правил|	✓|	E → E + O, O → O * O|
|Вложенные структуры|	✓	|E → ( E ) — скобочные выражения|
### Примеры правил, доказывающих КС-грамматику:
|Правило |	Обоснование |
|---------|------------|
|`E → E + O`	|Леворекурсивное правило, допустимое для КС-грамматик|
|`E → ( E )`	|Вложенная структура, невозможная в регулярных грамматиках|
|`P → id : T , P`|Рекурсивное правило для списка параметров|
## Метод анализа

### Выбранный метод: Рекурсивный спуск (Recursive Descent)

Метод рекурсивного спуска является реализацией нисходящего синтаксического анализа, где для каждого нетерминала грамматики создается своя функция, вызывающая другие функции для разбора вложенных конструкций.

### Алгоритм работы парсера (псевдокод):

```python
def parse_S():
    """S -> fn id ( P ) -> T { return E } ;"""
    
    # Проверка ключевого слова fn
    if not match('KEYWORD', 'fn'):
        error("Ожидалось 'fn'")
        synchronize()  # Метод Айронса
        return False
    
    # Проверка идентификатора
    if not match('IDENTIFIER'):
        error("Ожидался идентификатор")
        synchronize()
    
    # Проверка открывающей скобки
    if not match('LPAREN'):
        error("Ожидалась '('")
    
    # Разбор параметров (если есть)
    if current_token.type != 'RPAREN':
        parse_P()
    
    # Проверка закрывающей скобки
    if not match('RPAREN'):
        error("Ожидалась ')'")
    
    # Проверка стрелки ->
    if not match('MINUS') or not match('ARROW'):
        error("Ожидалось '->'")
    
    # Разбор типа возврата
    parse_T()
    
    # Проверка открывающей фигурной скобки
    if not match('LBRACE'):
        error("Ожидалась '{'")
    
    # Проверка ключевого слова return
    if not match('KEYWORD', 'return'):
        error("Ожидалось 'return'")
    
    # Разбор выражения
    parse_E()
    
    # Проверка закрывающей фигурной скобки
    if not match('RBRACE'):
        error("Ожидалась '}'")
    
    # Проверка точки с запятой
    if not match('SEMICOLON'):
        error("Ожидалась ';'")
    
    return True


def parse_E():
    """E -> id | num | ( E ) | E + E | E - E | E * E | E / E"""
    
    token = current_token()
    
    if token.type in ['IDENTIFIER', 'NUMBER']:
        next_token()  # Потребляем id или число
    
    elif token.type == 'LPAREN':
        next_token()  # Потребляем '('
        parse_E()      # Разбор выражения внутри скобок
        if not match('RPAREN'):
            error("Ожидалась ')'")
    
    else:
        error("Ожидалось выражение (идентификатор, число или скобки)")
        return
    
    # Обработка бинарных операторов
    if current_token().type in ['PLUS', 'MINUS', 'MULTIPLY', 'DIVIDE']:
        next_token()  # Потребляем оператор
        parse_E()      # Разбор правого операнда4

def _add_error(self, expected, found):
    """Добавляет ошибку в список (нейтрализация - анализ продолжается)"""
    token = self._current_token()
    if token:
        self.errors.append({
            'fragment': token['value'],
            'line': token['line'],
            'pos': token['start'],
            'message': f'Ожидалось "{expected}", найдено "{token["value"]}"'
        })
    else:
        self.errors.append({
            'fragment': 'конец строки',
            'line': 1,
            'pos': 1,
            'message': f'Ожидалось "{expected}", достигнут конец ввода'
        })
    # Анализ ПРОДОЛЖАЕТСЯ - ключевая особенность метода Айронса


def synchronize(self, follow_set):
    """Синхронизация: пропуск токенов до встречи с токеном из follow_set"""
    while self._current_token() and self._current_token().type not in follow_set:
        self._add_error("синхронизация", self._current_token().value)
        self.pos += 1
   ```

## Диагностика и нейтрализация ошибок методом Айронса
### Основные принципы метода
Метод Айронса (Irons, 1968) предназначен для обработки синтаксических ошибок при нисходящем разборе без возврата (backtracking).

Ключевая идея: При обнаружении синтаксической ошибки анализатор НЕ ОСТАНАВЛИВАЕТСЯ, а пропускает «проблемные» символы до тех пор, пока не сможет продолжить разбор с корректного места. Это позволяет обнаружить ВСЕ ошибки в программе за один проход.

### Два этапа обработки ошибки

## Синхронизирующие токены (множество FOLLOW)
Для каждого правила грамматики определены токены, с которых можно продолжить разбор после ошибки:
|Правило |	Синхронизирующие токены | Пояснение |
|---------|------------|----------|
|S (функция)	|`fun, EOF`|	При ошибке в функции пропускаем до следующей функции или конца файла|
|P (параметры)	|`), {`|	При ошибке в параметрах пропускаем до закрывающей скобки или начала тела|
|T (тип)	|`{, return, ;, }`|	При ошибке в типе пропускаем до начала тела или оператора|
|E (выражение)	|`;, }, ), +, -, *, /`|	При ошибке в выражении пропускаем до разделителя или оператора|
|Операнд	|`;, }, ), +, -, *, /, id, num, (`|	При ошибке в операнде пропускаем до корректного начала|
## Алгоритм match() с поддержкой метода Айронса
``` def match(self, expected_type, expected_value=None):
    """
    Сопоставляет текущий токен с ожидаемым.
    При ошибке: диагностика + нейтрализация (метод Айронса)
    """
    token = self._current_token()
    
    if not token:
        # ДИАГНОСТИКА: конец строки
        self._add_error(expected_type, 'EOF')
        return False
    
    if token['type'] == expected_type:
        if expected_value is None or token['value'] == expected_value:
            self.pos += 1  # УСПЕХ: переходим к следующему токену
            return True
    
    # ========== ДИАГНОСТИКА ==========
    self._add_error(expected_type, token['value'])
    
    # ========== НЕЙТРАЛИЗАЦИЯ ==========
    # Пропускаем токены до встречи с синхронизирующим
    follow_set = self._get_follow_set()
    while self._current_token() and self._current_token()['type'] not in follow_set:
        self.pos += 1  # пропускаем "проблемный" токен
    
    return False  # ошибка, но анализ ПРОДОЛЖАЕТСЯ
```
